## Hallazgos principales

### P0/P1 - Continuidad/versionado desalineados

- `pyproject.toml` dice version `0.35.13`.
- `src/flujo/version.py` dice version `0.35.13`.
- `python -m flujo version` muestra `0.35.13`.
- Pero hay handoffs de raiz que dicen `v0.40` / `v0.40.1` y commits recientes tambien sugieren eso.
- `context/LAST_HANDOFF.md`, que README declara como fuente de verdad, sigue con fecha 2026-06-24 y version `0.35.13`.
- `AGENTS.md` ya habla de estado 2026-06-28.

Riesgo: una IA o persona que retome el repo puede seguir instrucciones obsoletas o creer que una feature esta integrada de una forma distinta.

Recomendacion inmediata:

1. Decidir version real actual: mantener `0.35.13` o subir a `0.40.1`.
2. Actualizar en sincronía:
   - `pyproject.toml`
   - `src/flujo/version.py`
   - `context/LAST_HANDOFF.md`
   - `README.md` si menciona estado operativo
   - `AGENTS.md`
3. Dejar un solo handoff activo y archivar el resto.

### P1 - Comandos documentados no coinciden completamente

En `HANDOFF_2026-06-28_full.md` aparecen comandos como:

```bash
py -m flujo route where ...
py -m flujo index build ...
```

Pero la CLI real expone los addons bajo namespace `hub`:

```bash
py -m flujo hub route ...
py -m flujo hub index ...
```

`python -m flujo route --help` falla con `No such command 'route'`.

Recomendacion:

- O se registran alias top-level `route` e `index` compatibles.
- O se corrige la documentacion/handoff para usar siempre `py -m flujo hub route` y `py -m flujo hub index`.

### P1 - Indice de handoffs apunta a carpeta inexistente

`docs/handoffs/README.md` dice que los handoffs historicos estan en:

```txt
docs/handoffs/archive
```

Pero esa carpeta no existe en el checkout revisado.

Recomendacion:

- Crear `docs/handoffs/archive/` y mover ahi handoffs historicos.
- O corregir el indice para apuntar a la ubicacion real (`.archive/checkpoints/`, `checkpoints/`, raiz, etc.).

### P2 - Repo con mucha historia y artefactos mezclados

Conteo aproximado de archivos trackeados por area:

- `.archive`: 225
- `projects`: 95
- `docs`: 93
- `src`: 76
- `scripts`: 72
- `tools`: 36
- `tests`: 34
- `svg`: 34
- `checkpoints`: 19

Hay binarios/artefactos grandes trackeados en `datadrops/` y `svg/`, por ejemplo:

- `datadrops/.../rave_edit_rd_real_v5.png` ~1.6 MB
- `datadrops/.../suplementos3d.jpg` ~1.1 MB
- varios SVG vectorizados entre ~300 KB y ~730 KB

No es un problema grave por tamaño actual (~14 MB sin `.git`), pero conviene definir politica: que se versiona, que se genera, que se archiva.

### P2 - Makefile usa `python3` por defecto, pero reglas operativas piden `py`

`Makefile` usa:

```make
PYTHON ?= python3
```

Mientras `README`, `AGENTS` y `LAST_HANDOFF` enfatizan Windows/Git Bash con `py`.

Recomendacion:

- Documentar: `make test PYTHON=py` en Windows.
- O ajustar Makefile para detectar/permitir `PYTHON ?= py` en contexto Windows.

### P3 - `src/flujo/cli.py` es muy grande

`src/flujo/cli.py` pesa ~89 KB y concentra muchos comandos. Funciona, pero a futuro sera mas facil mantenerlo si se separa por subapps:

- `cli_jobs.py`
- `cli_intake.py`
- `cli_render.py`
- `cli_hub.py`
- `cli_airdrop.py`

No es urgente porque tests pasan.
